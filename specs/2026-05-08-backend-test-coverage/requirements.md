# Requirements: Backend And Progress Test Coverage

## Scope

Add automated tests for the current Spira MVP contracts after the backend foundation and frontend/backend integration work.

This task covers backend unit tests, backend Spring integration tests, and a small frontend unit test layer for the shared progress contract.

## Included

- Unit tests for backend target progress behavior.
- Unit tests for backend resource type validation.
- Unit tests for backend goal option service rules.
- Unit tests for backend reality kind normalization and grouping.
- Spring Boot GraphQL integration tests using Spring test support.
- H2-backed persistence for integration tests.
- Frontend unit tests for `targetProgress` and `goalProgress`.
- Documentation explaining what tests were created, why they exist, and how to run them.
- GitHub Actions CI that runs frontend and backend validation on push, pull requests, manual dispatch, and scheduled execution.
- Test results reporting through Allure report artifacts generated from backend test runs.

## Excluded

- Authentication and authorization tests, because auth is intentionally not implemented yet.
- AI orchestration tests, because AI persistence and proposals are not implemented yet.
- GROW session tests, because GROW sessions are not implemented yet.
- Playwright E2E tests, because this task focuses on unit and integration coverage.
- PostgreSQL/Testcontainers migration tests, because the current local integration profile uses H2 and production migrations include PostgreSQL-specific SQL.

## Product Contracts To Protect

- Goals are structured workspaces containing reality, options, resources, and targets.
- Progress is calculated only from targets.
- All targets contribute equally to goal progress.
- Numeric progress matches frontend behavior, including inferred starts, reverse direction, and clamping.
- Binary progress is `1` when done and `0` when not done.
- Checklist progress is completed items divided by total items.
- A goal with no targets has progress `0`.
- The backend resource type is `email`, not the old `contact` alias.
- Resource labels are capped at 20 characters, with autogenerated labels for links and emails when possible.
- Options keep deterministic positions and one selected option per goal.
- Reality accepts action/obstacle singular and plural inputs but stores canonical plural kinds.
- The no-auth backend remains a single-user persistence API until auth work is added later.

## Success Criteria

- `npm.cmd test` passes.
- `cd backend && .\mvnw.cmd test` passes.
- Backend integration tests exercise the real Spring GraphQL controller/service/repository path.
- Test documentation exists under `docs/`.
- CI workflow exists under `.github/workflows/` and runs frontend/backend validation automatically.
- CI publishes backend test report artifacts, including Allure output.
- This spec exists under `specs/`.
